# [[query-variations]] — Per-engine × Per-language Variation Reference (Research-Gated)

**Status:** Research-gated reference (Perplexity pass, 2026-07-13).
**Scope:** focused on the **established 6** languages — ZH, RU, JA, KO, ES, PT — and their engines (Google, Bing, Yandex, Sogou/Haosou, Naver). Planned languages (FR, DE, CS, PL, UK, AR, HI) are deferred — see PRD §7.5; they require operator discovery + corpus evidence before use.
**Rule:** the planner may use a variation feature only when the relevant `(engine, language, platform)` lock is documented here with a source **and** (where applicable) integration-tested against that engine. No feature may be emitted as a rule without a `[VERIFIED]` or `[PARTIALLY VERIFIED]` tag.

## Evidence labels
- **[VERIFIED]** — directly supported by a primary engine document or specific source.
- **[PARTIALLY VERIFIED]** — broad capability supported, but asserted syntax / recall effect / cross-language generalization is not.
- **[UNVERIFIED—DO NOT EMIT]** — plausible hypothesis, no admissible evidence; do not emit as a rule.
- **[CONTRADICTED]** — conflicts with available documentation / overstates evidence.

The prior draft was a useful *specification* but was **not** evidence that its operators, idioms, or recall effects work. This revision gates every claim.

## 1. Two-layer model

### 1.1 Layer 1 — Stance slots (language-neutral)
The 8 slots are sound **product-design dimensions**, but the available material does **not** establish them as an empirically validated, cross-culturally complete taxonomy. Treat them as *optional query angles*, not cultural facts.

**Live A/B test (Naver, Korean, 2026-07-14):** A formal query ("두통 치료 약물 임상 시험" = headache treatment drug clinical trial) and a personal-experience query ("내 두통 약이 효과가 없었어" = my headache medicine didn't work) returned **different result sets** — personal experience returned ~50KB more content than the formal variant. This is early empirical evidence that stance variation changes which documents are surfaced. Systematic evaluation per (language, engine, platform, domain) tuple is still needed (see §6).

> **Note on [PARTIALLY VERIFIED] statuses in this section:** The phenomenon exists (confirmed by literature and live A/B test), but the specific 8-slot set is a product-design hypothesis, not a proven universal taxonomy. The [PARTIALLY] reflects the inherent complexity of language — there are too many dialects, registers, and cultural stance patterns for any fixed slot set to be "complete" across all languages. This is a design constraint, not a defect. Build with what's known, measure, and iterate.

## 3. Temporal handling
### 3.1 Intent & content markers — [PARTIALLY VERIFIED]
> **Note:** Temporal intent detection is a real IR task (literature-supported). The [PARTIALLY] reflects the inherent complexity of language — no study has proven that 最近/недавно/최근/reciente/recente all function as uniform recency signals across languages. Different languages grammaticalize time differently. Treat these as *potential* signals, not guaranteed classifiers. This is a design constraint, not a defect.

Treat 最近 / недавно / 최근 / reciente / recente as *potential* temporal signals; don't infer universal recency intent without context. Extract in result content as document-language text, resolve relative to document date. **Never** interpret a relative temporal word as a 2026 timestamp without document context.

### 3.2 Document-relative time — [VERIFIED as IR rule]
Relative expressions in archived docs resolve against that doc's publication/thread/archive-capture date, not today. Precedence: publication > thread-post > archive-capture > crawl/index. Store raw expression + normalized interval + confidence.

### 3.3 Era / event anchors — [PARTIALLY VERIFIED]
> **Note:** Era/event-aware retrieval is a real temporal IR research area (Sasaki 2025, Berendt 2012). The [PARTIALLY] reflects the inherent complexity of calendar systems — no search engine natively understands 令和/平成/昭和 or Hijri. The concept is verified; the engine-operator implementation requires manual date-range conversion. This is a design constraint, not a defect.

| Anchor            | Status                             | Rule                                                                                                                |
| ----------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| JA 令和/平成/昭和       | [PARTIALLY VERIFIED]               | Versioned era→Gregorian map; engine doesn't understand eras natively.                                               |
| ZH 朝代 / 非典时期 / 文革 | [PARTIALLY VERIFIED]               | Semantic/event expansion → date interval with uncertainty; don't present one event year as full discourse interval. |
| Hijri             | [PARTIALLY VERIFIED]               | Convert + expand range; no engine offers Hijri-native filtering without direct doc.                                 |
| CS / PL / UK      | [UNVERIFIED—needs corpus research] | No reusable policy yet.                                                                                             |


### 3.4 "Emerging" — [VERIFIED as product policy, not cultural fact]
Soft ranking feature, not hard 5-yr exclusion. Signals: recent pub/discovery date, recent corpus-frequency rise, explicit novelty language, recent event/version. Calibrate per corpus/topic; measure, don't assume (no validated basis for "Western FDA cadence / TCM timescale / RU preprint rhythm" as universal rules).
