# Prowl v0.1.0 — Implementation Roadmap

> Strict dependency-ordered feature roadmap from the current (near-empty) TypeScript
> monorepo to a usable local, Pi-first vertical slice — **release target: Prowl v0.1.0**.
> v0.1.0 is the **`/prowl search` vertical slice** only (snippets + optional `--read`
> evidence mode). Everything else — `/prowl query` / `/prowl chat`, persistent
> `StoragePort`/`CatalogPort`, negative-search/exclusion, multilingual dork-planner
> expansion, RAG/wiki/deep-dive/loops, web/CLI adapters, browser-interaction automation,
> and public deployment — is **deferred after v0.1.0** (see §5).
> **Roadmap only.** This document proposes phases; it writes no code, edits no Docker
> config, installs no packages, and commits nothing.

## Workflow-tag legend (rpiv-pi canonical meanings)

These tags choose the agent workflow to use when a phase is later executed — they do
**not** describe whether a phase has live I/O, whether it is deployed, or whether it
writes ports/types.

- **ship** — a small, low-risk, well-understood change. Fast path; it skips research
  and review. Use for isolated config, docs, or obvious glue after the design is settled.
- **build** — normal feature work. Use for an adapter, a pipeline slice, command
  behavior, or anything needing research/design, implementation, validation, and a review loop.
- **arch** — a consequential architectural or cross-cutting change affecting multiple
  layers, contracts, dependency direction, or a subsystem boundary.

## 2. Guiding Constraints (from PRD / guidance)

- **Hexagonal:** Core = ports + orchestration + ranking/policy. Zero runtime deps; never
  imports Pi, Docker, `fetch`, or terminal APIs. Adapters own I/O and Pi UI.
- **SearXNG = discovery/ranking** (candidate URLs). **Firecrawl = conditional extraction**
  of a *bounded selected set* only — never scrape every result by default.
- **v0.1.0 = the search slice, local Pi-first, single-user.** v0.1.0 ships only
  `prowl search` (snippets default; `--read` evidence mode gated behind a flag). It does
  **not** ship `query`/`chat`, persistent `StoragePort`/`CatalogPort`, negative-search,
  multilingual dork expansion, RAG/wiki/deep-dive/loops, web/CLI adapters, browser
  automation, or public deployment — those are deferred (§5). Keep the `prowl-core` +
  `prowl-pi` split; it is the correct structure and is not changed by this roadmap.
- **Firecrawl is conditional with an explicit user trigger:** `prowl search --read <topic>`
  enables evidence mode. Default `search` uses snippets only. Read mode selects a bounded
  set (3–5 diverse, high-value URLs) and returns Markdown for synthesis.
- **Single model (Qwen)** for both planning and synthesis, via OpenAI-compatible endpoint.
- **Negative-search / exclusion (PRD §8.4)** is **deferred after v0.1.0** — explicitly out
  of the v0.1.0 gate (see §5), no longer "post-MVP hardening within the first roadmap."

### Phase 3 — `ModelPort` (Qwen via OpenAI SDK)  ·  **build**

**Goal / why now:** The single model is needed by `PLAN` (query planning) and
`SYNTHESIZE` in the pipeline (Phase 5). Building it as an isolated adapter early keeps the
pipeline phase free of provider concerns.

**Files:**
- `packages/pi/src/model-client.ts` (NEW) —
  ```ts
  import OpenAI from "openai";
  import type { ModelPort } from "prowl-core";

  const client = new OpenAI({
    apiKey: process.env.PROWL_MODEL_API_KEY ?? "",
    baseURL: process.env.PROWL_MODEL_BASE_URL, // OpenAI-compatible (Venice/OpenRouter/etc.)
  });
  const MODEL = process.env.PROWL_MODEL ?? "qwen3.6-plus";

  export const modelClient: ModelPort = {
    async generate(prompt: string, opts?: Record<string, unknown>) {
      const res = await client.chat.completions.create({
        model: (opts?.model as string) ?? MODEL,
        messages: [{ role: "user", content: prompt }],
      });
      return res.choices[0]?.message?.content ?? "";
    },
  };
  ```
- `packages/pi/src/index.ts` (MODIFY) — export `modelClient`.
- `.env.example` (MODIFY) — add `PROWL_MODEL_API_KEY`, `PROWL_MODEL_BASE_URL`,
  `PROWL_MODEL` under a new `## Prowl model` section (per env-config conventions).

**Dependencies:** #0.

**Acceptance criteria:**
- `bunx tsc --noEmit -p packages/pi/tsconfig.json` passes.
- `modelClient.generate("ping")` returns a non-empty string when `PROWL_MODEL_API_KEY` +
  `PROWL_MODEL_BASE_URL` point at a live Qwen-compatible endpoint.
- `grep -n "modelClient" packages/pi/src/index.ts` confirms the export.

**Out of scope:** multi-model routing, DSPy, tiered model selection (deferred, §5 D6).

### Phase 5 — Core `search` pipeline (snippets-only)  ·  **arch**

**Goal / why now:** The orchestration heart for the slice — composes ports into the
`search` command per `PRD §2.1`: `PLAN → SCATTER → GATHER → SYNTHESIZE → PRESENT`.
Snippets-only (no Firecrawl). This is the central architectural seam (hence **arch**): it
owns the command flow + ranking/selection policy, lives entirely in core with zero I/O, and
gates the demo (Phase 6) and every later command.

**Files:**
- `packages/core/src/pipeline.ts` (NEW) — primitive functions used by `search`:
  - `plan(query, model)` → `PLAN` (returns a (re)formed query / query-set via `ModelPort`).
  - `scatter(search, querySet, engines?)` → `SCATTER` (calls `SearchPort`, returns
    `SearchResult[]`).
  - `gather(results)` → `GATHER` (dedupe/normalize `SearchResult[]`).
  - `synthesize(model, query, gathered)` → `SYNTHESIZE` (calls `ModelPort` with snippets).
  - `present(presenter, content)` → `PRESENT`.
- `packages/core/src/commands.ts` (NEW) — `search(input, deps)` composer:
  `PLAN → SCATTER → GATHER → SYNTHESIZE → PRESENT`. Accepts a `deps` object
  (`{ search: SearchPort; model: ModelPort; present: PresenterPort }`) — structural
  injection; adapters supply concretes at call time. No port implementation lives here.
- `packages/core/src/ranking.ts` (NEW) — snippet-level selection/dedup policy used by
  `GATHER`/`SYNTHESIZE` context building.
- `packages/core/src/index.ts` (MODIFY) — export `pipeline`, `commands` functions/types.

**Dependencies:** #1 (contracts), #3 (model available to type, core only types it).

**Acceptance criteria:**
- `bunx tsc --noEmit -p packages/core/tsconfig.json` passes with **zero** imports of
  `fetch`, `process`, pi, or any adapter (`grep -rn "fetch\|process.env" packages/core`
  returns nothing).
- A unit test (Phase 8) drives `search` with **mock** ports and asserts the primitive
  sequence `PLAN→SCATTER→GATHER→SYNTHESIZE→PRESENT` + output shape (summary + sources).
- `grep -n "export" packages/core/src/index.ts` includes the new composers.

**Out of scope:** Firecrawl wiring (Phase 7); `query`/`chat` composers (deferred, §5 D1/D2);
Pi UI (Phase 4/6 adapter side).

**Workflow tag:** **arch** (subsystem boundary + orchestration owning core policy).

## 5. Deferred after v0.1.0

Everything below is **explicitly out of scope for v0.1.0** (the `/prowl search` slice).
These items were previously scattered as `--` rows and post-MVP notes; they are collected
here so the v0.1.0 gate is unambiguous. They become future phases after the slice is stable
and are tracked in `docs/planning/prowl-versions.md` (v0.2+). `prowl-core` + `prowl-pi`
remain the correct package structure for all of them.

### D1 — `/prowl query` command
**What:** `READ → SYNTHESIZE → PRESENT` over prior results / a focused question.
**Why deferred:** needs last-results context (`StoragePort` impl, D3) and `UserPromptPort`
(`ask_user` / follow-up). Not required to prove the v0.1.0 search slice.

### D2 — `/prowl chat` command
**What:** `PLAN/REFLECT → (SCATTER·GATHER·READ) → SYNTHESIZE → PRESENT → REFLECT` loop.
**Why deferred:** needs `StoragePort` (D3) + `UserPromptPort` (`REFLECT`/`ask_user`), the
heaviest adapter surface. Out of v0.1.0.

### D3 — Persistent `StoragePort` / `CatalogPort` (fs) implementations
**What:** tenant-scoped filesystem implementations of the contracts defined in Phase 1
(`~/.prowl/...`), plus the site catalog (`CatalogPort` + `SiteEntry`).
**Why deferred:** v0.1.0 is local, single-user, snippets-only — the slice needs only
`SearchPort` + `ModelPort` + `PresenterPort`. fs persistence only becomes necessary when
`query`/`chat` need last-results context (D1/D2) and negative-search needs a catalog seed
(D4). Defining the contracts now (Phase 1) keeps the tenant seam without delaying search.

### D4 — Negative-search / exclusion (PRD §8.4)
**What:** before litter-web search, query SEO engines (Google/Bing) to build a per-topic
exclusion list, applied to subsequent searches. Whitelist (wikipedia/pubmed/arxiv) survives.
**Why deferred:** not required to prove v0.1.0 (snippets → synthesis → output). Adds
classification / whitelist / global-vs-topic exclusion policy that can stall usable search.
Becomes a post-v0.1.0 phase once `search` is stable and `CatalogPort` exists (D3).
**Planned shape:** core policy (`buildExclusion` querying SEO engines via `SearchPort`,
post-filter + `-site:` dork builder) + `CatalogPort` seed list + composer hook.

### D5 — Multilingual dork-planner expansion
**What:** the full 8–12 query-variation generator (`dork-planner.ts`) across the core 6
languages (ZH, RU, JA, KO, ES, PT).
**Why deferred:** v0.1.0 `PLAN` is a single-language planning stub (Qwen reforms the user
query), not the multi-language emitter. Expanding the planner is a later hardening phase,
not a v0.1.0 gate.

### D6 — RAG / wiki / deep-dive / bounded loops
**What:** `deep-dive` mode + Karpathy-style wikis, pi-remembers RAG (cross-topic memory),
bounded research loops (`/prowl loop`) with tiered model routing, DSPy for structured
prompting/composition, and the full primitive set (`EVALUATE`, `PERSIST`, `ADD_SITE`,
`LINT`).
**Why deferred:** none of this is needed for the search slice; it is the v0.2+ research
depth work tracked in `prowl-versions.md`.

### D7 — Web adapter (`prowl-web`) + CLI adapter (`prowl-cli`)
**What:** a static web UI (`prowl.moe`) and a standalone CLI binary for non-pi users.
**Why deferred:** v0.1.0 is Pi-first and local by design. Public-facing adapters come only
after the local slice is proven, and pair with D9 (public deployment).

### D8 — Browser-interaction automation
**What:** driving a real browser to interact with pages (beyond Firecrawl's headless
extraction, which remains a v0.1.0 conditional path).
**Why deferred:** explicitly out of v0.1.0 scope per the project's bounded-v0.1 stance.
Firecrawl's own Playwright/Chromium is still used for `--read` extraction (Phase 7); that
is extraction, not Prowl-driven browser automation.

### D9 — Public deployment (`prowl.moe`, VPS, marketplace)
**What:** hosting SearXNG + Firecrawl on a public VPS (e.g., RackNerd), buying the domain,
a marketplace for sellable wikis, etc.
**Why deferred:** v0.1.0 is an internal, local, single-user EccoMuse research tool. Public
deployment is a v1.0+ concern (see `prowl-versions.md`).

## 6. Recommendations & Demo

### Recommended first implementation phases
**Phase 0** (ship, 5 min) → **Phase 1** (arch, contract-only) → **Phase 2** (build, Pi
spike) → **Phase 3** (build, ModelPort) → **Phase 4** (build, PresenterPort + wiring) →
**Phase 5** (arch, pipeline) → **Phase 6** (build, first demo) → **Phase 7** (build, `--read`)
→ **Phase 8** (build, tests + acceptance). This is the shortest path to a shippable
`prowl search` and defines v0.1.0.

### Smallest user-visible end-to-end demo (Phase 6, the v0.1.0 gate)
```
/prowl search "magnesium orotate personal experiences"
  → SearXNG result snippets
  → Qwen synthesis with URLs
  → rendered Pi output
```
No `query`/`chat` required for this demo. `--read` evidence mode (Phase 7) is an additive
flag on top of the same command and is part of v0.1.0.

### After v0.1.0 (deferred — see §5)
`query` / `chat`, persistent `StoragePort`/`CatalogPort`, negative-search, multilingual
dork expansion, RAG/wiki/deep-dive/loops, web/CLI adapters, browser automation, and public
deployment are all post-v0.1.0. They are sequenced in `docs/planning/prowl-versions.md`
starting at v0.2.

## 7. Assumptions & Open Questions (resolve before implementation)

1. **Pi extension registration API.** The exact `registerCommand` / `present` / `ask_user`
   host-surface signatures are assumed from `.rpiv/guidance/packages/pi/src/architecture.md`.
   **Must confirm** against the live pi extension SDK before Phase 2 & 4 — the host API
   shape determines the spike (Phase 2) and `pi-ports.ts` (Phase 4). This is the explicit
   unknown that makes Phase 2 a **build**, not a ship.
2. **Model endpoint & key.** Qwen assumed reachable via an OpenAI-compatible base URL
   (`PROWL_MODEL_BASE_URL`) with `PROWL_MODEL_API_KEY`. **Confirm** the exact model id
   (`qwen3.6-plus`?) and that the endpoint supports the chat-completions shape used in #3.
3. **`--read` trigger + selection.** Assumed flag `prowl search --read <topic>`; bounded
   set default 3–5, diverse across engines/domains, high authority tier. **Confirm** the
   flag name and the selection heuristic (authority-tier vs. user `ask_user` pick vs. top-k
   by snippet signal) for Phase 7.
4. **`PLAN` output shape (v0.1.0).** `plan()` is a **single-language planning stub** (Qwen
   reforms the user query) — the full multi-language/dork variation generator (`dork-planner.ts`,
   8–12 variations) is **deferred** (§5 D5). Confirm v0.1.0 `PLAN` is the stub, not the
   multi-language emitter.
5. **Build/run tooling.** Assumed Bun (`bun test`, `bunx tsc`). **Confirm** `tsc` is the
   intended type-check (peer dep `typescript@^7`) vs. another driver, and that `bun test` is
   the test runner.
6. **Deploy mechanism.** Assumed a copy step from `packages/pi` →
   `~/.pi/agent/extensions/prowl/`. **Confirm** the pi extension loader expects source `.ts`
   directly or a build artifact, and whether the `exports` map (Phase 0) satisfies it.
7. **PresenterPort host surface.** Assumed the spike (Phase 2) discovers a `present(message)`
   host call. **Confirm** whether rendering is a function, a returned value, or a streamed
   message — affects `pi-ports.ts` (Phase 4) shape.
